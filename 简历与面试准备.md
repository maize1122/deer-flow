- **8002**：Provisioner（仅部分沙箱/K8s 配置下启用）

## 三、项目搭建流程（从零到可访问）

### 3.1 通用前置

1. 克隆仓库：`git clone https://github.com/bytedance/deer-flow.git && cd deer-flow`  
2. 生成配置：`make config`（从 `config.example.yaml` 等生成 **`config.yaml`**，若已存在会中止，避免覆盖）  
3. 编辑 **`config.yaml`**：在 `models:` 下至少配置 **一个模型**（`use` 指向 LangChain 类或项目内自定义 Provider）  
4. 在 **`.env`** 或 shell 中填入 `config.yaml` 里引用的变量（如 `$OPENAI_API_KEY`、搜索类 `TAVILY_API_KEY`、`INFOQUEST_API_KEY` 等）**不要**把密钥写进简历或公开仓库  

### 3.2 路径 A：Docker（官方推荐）

1. 确认 Docker 可用：`docker info`  
2. `make docker-init`：拉取沙箱等相关镜像（按 `config.yaml` 中 sandbox 镜像或默认镜像）  
3. `make docker-start`：按 **`config.yaml` 中 sandbox 模式** 决定是否起 provisioner  
4. 浏览器访问：**http://localhost:2026**  

国内网络可提前设置镜像（示例）：`UV_INDEX_URL`、`NPM_REGISTRY` 后再执行 docker 相关命令。

### 3.3 路径 B：本机开发

1. `make check`：检查 **Node 22+**、**pnpm**、**uv**、**nginx** 等  
2. `make install`：`backend` 下 `uv sync`，`frontend` 下 `pnpm install`  
3. （可选）`make setup-sandbox`：预拉沙箱镜像  
4. （可选）`python scripts/load_memory_sample.py`：加载示例记忆，便于测「设置 - 记忆」  
5. `make dev`：启动 LangGraph + Gateway + Frontend + Nginx（带热更新）  
6. 访问：**http://localhost:2026**  

### 3.4 配置升级

- 官方升级配置字段：`make config-upgrade`（把 `config.example.yaml` 新增字段合并进现有 `config.yaml`）  

### 3.5 生产向 Docker

- `make up` / `make down`：构建并启动或停止生产组合容器（仍常通过 **2026** 访问，具体以仓库文档为准）

## 四、功能全貌（按模块讲清楚）

### 4.1 用户侧：Web 控制台

- 对话、流式输出、模型切换、计划模式、子智能体开关、思考链（视模型能力）等运行时选项（具体 UI 以当前版本为准）。  
- 技能管理、MCP 配置、记忆查看与配置、线程与上传文件、产物（artifacts）浏览等通过 **Gateway** 与 **LangGraph** 配合完成。

### 4.2 智能体运行时（LangGraph）

- **单一注册的 Lead Agent**：入口在 `langgraph.json`，工厂函数 `make_lead_agent`。  
- **ThreadState**：在通用 Agent 状态上扩展沙箱句柄、线程数据、标题、产物、todos、上传文件、已查看图片等；部分字段用 **reducer** 做合并/去重。  
- **可配置项（configurable）**：如 `model_name`、`thinking_enabled`、`is_plan_mode`、`subagent_enabled` 等，影响中间件与工具集。

### 4.3 中间件链（顺序敏感）

典型顺序包括：线程数据与目录隔离 → 上传注入 → 沙箱就绪 →（可选）摘要 →（可选）计划 Todo → 标题 → 记忆入队 →（条件）看图 → **澄清中断（通常最后）**。  
**澄清中间件放最后** 的原因：要在下游工具触发「需要用户补充信息」时，能 **interrupt** 图执行，避免白跑。

### 4.4 技能（Skills）

- 以 **`SKILL.md`** 为主的模块化能力说明，可带脚本与资源；放在 `skills/public`（仓库内建）与 `skills/custom`（用户自定义）。  
- **渐进加载**：按任务需要注入，控制上下文体积。  
- 可通过网关安装 **`.skill` 包**；支持标准 frontmatter（version、author、compatibility 等）。

### 4.5 工具生态

- **沙箱工具**：`ls`、读写、`str_replace`、`bash`（本地 Provider 下 bash 默认关闭，容器沙箱更安全）。  
- **内置工具**：如展示文件、请求澄清、看图、`task` 子智能体等。  
- **社区工具**：Tavily、Jina、Firecrawl、图片搜索等（以 `config.yaml` 启用为准）。  
- **MCP**：把外部工具以 MCP 协议接入，支持多种传输与 OAuth。

### 4.6 子智能体（Sub-agents）

- Lead Agent 通过 **`task` 类工具** 委派子任务；线程池异步跑，带超时与并发上限（文档中常见如每轮最多 3 个、超时约 15 分钟量级，以代码配置为准）。  
- 子智能体上下文与主会话 **隔离**，减少互相干扰。  
- 结果回传主智能体汇总。

### 4.7 沙箱与文件布局（逻辑路径）

- 每线程隔离：`uploads`、`workspace`、`outputs` 映射到主机或容器内目录。  
- 技能挂载在 **`/mnt/skills`**，与仓库 `skills` 目录对应。  
- Provider 包括本地与 **AioSandbox（Docker）** 等；高阶场景可通过 **Provisioner + K8s**。

### 4.8 长期记忆（Memory）

- 会话结束后或过程中由 **LLM 抽取** 用户画像、事实、偏好等，写入本地 JSON（带缓存失效策略）。  
- 通过 **debounce** 合并更新，减少调用次数。  
- 注入系统提示时带 **置信度较高的事实** 与上下文摘要类信息。  
- 应用层会去重，避免重复事实无限膨胀。

### 4.9 Gateway API（FastAPI）主要职责

- 模型列表、MCP 与技能 CRUD/安装、记忆读写与配置、线程上传与列表、产物下载、建议问题、线程本地数据清理（`DELETE /api/threads/{id}`）等。  
- 与 **嵌入式 DeerFlowClient** 的返回结构在 CI 中对齐，避免 HTTP 与库两套接口漂移。

### 4.10 IM 通道

- **飞书 / Slack / Telegram**：长连接或 Socket Mode 等，无需公网 IP 即可收消息（按各平台文档配置）。  
- 通过 `config.yaml` 的 `channels` 配置 `langgraph_url`、`gateway_url`；**Docker 内**需用服务名（如 `http://langgraph:2024`）而非 `localhost`。  
- 常用命令：`/new`、`/status`、`/models`、`/memory`、`/help`。

### 4.11 可观测与安全

- **LangSmith**：环境变量开启后追踪 LLM、工具、运行轨迹。  
- **安全提示**：默认面向 **本机可信环境**；公网部署需鉴权、IP 白名单、网络隔离等。产物下载对若干 Web 内容类型强制附件，降低 XSS 面。

### 4.12 嵌入式使用

- 不启 HTTP 也可 `from deerflow.client import DeerFlowClient` 在同进程调用与网关一致的接口（chat、stream、list_models 等）。

## 五、面试官可能问的问题 + 八股（含参考答案）

### 5.1 项目理解类

**Q1：DeerFlow 和「做一个 ChatGPT 网页」有什么区别？**  
**A：** 不仅是聊天 UI。DeerFlow 提供 **有状态图运行时（LangGraph）**、**每线程沙箱与文件系统**、**可插拔工具链（含 MCP）**、**子智能体并行**、**技能驱动的流程知识**，以及 **跨会话记忆**；网关负责 **模型/技能/MCP/上传/记忆** 等资源管理，和纯「调一个 completion API」的产品层级不同。

**Q2：为什么要有 Gateway，而不是前端直接打 LangGraph？**  
**A：** 职责分离：**LangGraph** 专注 **对话与工具调用状态机**；**Gateway** 承载 **与图无关或弱耦合的管理面能力**（模型元数据、MCP/技能配置、文件上传转换、记忆持久化、产物静态服务、线程清理等）。这样安全策略、鉴权、限流、审计以后也更好加在网关层。

**Q3：Harness 和 App 为什么要拆分？**  
**A：** **Harness**（`deerflow`）是可复用的智能体框架；**App**（`app.gateway`、`app.channels`）是具体产品部署与协议适配。**单向依赖**（App → Harness）+ **CI 边界测试**，避免框架反向依赖业务代码，便于单独发版、开源复用与单元测试。

**Q4：中间件链里「澄清」为什么放在最后？**  
**A：** 澄清往往对应 **人机协同中断**：需要 **interrupt** 当前 run 等待用户输入。若放太早，后续中间件或工具逻辑仍可能继续推进，导致状态不一致；放在最后能 **稳定截断** 执行流。

### 5.2 LangGraph / LangChain

**Q5：LangGraph 和 LangChain 的 LCEL 链有什么本质不同？**  
**A：** **LCEL** 更偏 **DAG/管道式** 组合；**LangGraph** 显式建模 **状态图**：节点、边、条件分支、**循环**、**人工介入（interrupt）**、**检查点（checkpoint）**，适合 **多步工具调用、重试、子图、持久化线程**。DeerFlow 的 Lead Agent 是长时间运行、带工具与记忆的，更适合图范式。

**Q6：什么是 StateGraph 里的 reducer？你们项目里怎么用？**  
**A：** 多分支或并发更新同一字段时，LangGraph 需要知道如何 **合并** 更新。**Reducer** 就是「把旧状态和新 patch 合成最终状态」的函数。DeerFlow 对 artifacts、viewed_images 等用自定义 reducer 做 **去重/合并**，避免并发工具结果互相覆盖或重复爆炸。

**Q7：ReAct 是什么？和 DeerFlow 的关系？**  
**A：** **ReAct** 是「推理 Thought + 行动 Action（常是工具）」交替的范式。现代 LLM Agent 常在聊天模板里用 **工具调用（tool calling）** 实现类似循环：模型决定调哪个工具 → 运行工具 → 把 observation 塞回上下文 → 再生成。DeerFlow 的 Lead Agent 即在图里 orchestrate 这一套，并加上沙箱、子智能体、记忆等扩展。

### 5.3 MCP、工具与沙箱

**Q8：MCP（Model Context Protocol）解决什么问题？**  
**A：** 把 **工具/资源/提示** 以 **标准协议** 暴露给模型宿主，实现 **进程隔离、语言无关、可插拔**。DeerFlow 可把 MCP Server 配进运行时，和内置工具一起给模型选用。

**Q9：为什么本地沙箱默认关 bash，而 Docker 沙箱才开？**  
**A：** 本地进程 **与宿主共享内核与用户权限**，bash 属于 **高危险面**（删文件、读隐私、起恶意进程）。容器沙箱提供 **更强隔离边界**（仍需镜像与 capability 加固）。因此默认策略是：**不可信执行走容器**；本地仅信环境再开 bash。

**Q10：Docker 与虚拟机在隔离上的区别？**  
**A：** **VM** 虚拟化硬件，跑完整 Guest OS，隔离强、资源重。**容器** 共享宿主内核，通过 **namespace + cgroups** 隔离进程视图与资源配额，启动快、密度高，但 **内核级漏洞** 可能影响同宿主容器。沙箱选型要在安全、性能、运维成本间权衡。

### 5.4 工程与网络

**Q11：Nginx 在这里做什么？**  
**A：** **统一入口**：TLS 终止（若配置）、静态前端、**路径分流**——例如 `/api/langgraph` 反代 LangGraph，`/api` 其他反代 Gateway，页面走 Next。对用户只暴露 **2026**，简化 CORS 与部署。

**Q12：SSE（Server-Sent Events）和 WebSocket 区别？**  
**A：** **SSE** 是 **单向** 服务器→浏览器的 **HTTP 长连接**（`text/event-stream`），基于标准 HTTP，自动重连较简单，适合 **模型流式 token**。**WebSocket** **全双工**，适合高频双向交互（协作编辑、游戏）。DeerFlow 流式侧常见 SSE 方案（具体以 SDK/路由实现为准）。

**Q5.13：FastAPI 相对 Flask 的亮点？**  
**A：** 原生 **async**、`asyncio` 生态友好；**Pydantic** 做请求响应校验；自动生成 **OpenAPI**；依赖注入清晰。网关场景下并发 IO（调模型、存文件）用 async 更容易榨干吞吐。

### 5.5 前端（Next.js）

**Q14：Next.js App Router 相对传统 SPA 的好处？**  
**A：** 服务端组件、布局与流式渲染、与元数据/SEO 更友好；可把一些请求放到服务端做，减少浏览器密钥暴露。DeerFlow 前端还需与 **LangGraph SDK**、**TanStack Query** 管理服务端状态与缓存。

**Q15：TanStack Query 解决什么问题？**  
**A：** 对 **服务端状态** 做缓存、去重、后台刷新、分页/无限滚动、错误重试；避免手写大量 `useEffect + fetch` 导致竞态与陈旧数据。

### 5.6 记忆 vs RAG

**Q16：DeerFlow 的记忆和 RAG 有什么差别？**  
**A：** **记忆** 这里是 **结构化/profile + 事实列表**，由对话 **抽取写入**，偏「用户长期偏好与画像」。**RAG** 通常是 **检索外部文档块** 注入上下文，偏「知识库问答」。二者可互补：记忆不是替代向量库；若要做企业知识库，往往还要加嵌入与检索管道。

**Q17：如何避免记忆无限膨胀？**  
**A：** 项目里有 **去重** 与 **摘要/裁剪策略**（具体阈值看配置与实现）；产品侧还可限制条数、按时间衰减、人工审核记忆条目。

### 5.7 子智能体与并发

**Q18：子智能体和「一次 tool call 里顺序调多个 API」区别？**  
**A：** 子智能体是 **独立上下文与独立推理循环** 的「小代理」，可 **并行**、可有 **不同工具子集**、有 **自己的终止条件**；适合大任务拆解。顺序 API 调用仍是 **同一上下文** 下的步骤，耦合度高、难并行扩展。

**Q19：线程池开多少、如何避免拖垮宿主？**  
**A：** 应有 **并发上限**、**超时**、**队列** 与 **背压**；子智能体文档中常见每轮并发上限与分钟级超时。生产还要配 **资源监控**、**限流**、**取消运行**。

### 5.8 安全与合规

**Q20：智能体执行 bash 的最大风险？**  
**A：** **数据泄露**（读敏感文件）、**破坏性命令**（`rm -rf`）、**供应链攻击**（下载恶意脚本）、**横向移动**（内网扫描）。缓解：**容器隔离**、**只读根文件系统**、**网络 egress 白名单**、**命令审计**、**最小权限**、**人工批准高危操作**。

**Q21：为什么产物下载要防 XSS？**  
**A：** 用户可能让模型生成 **HTML/SVG** 等；若浏览器 **inline 渲染** 且同源或共享 cookie，可能被植入脚本。**强制附件下载**或 **严格 CSP**、**隔离域名** 是常见防护。

### 5.9 部署与运维

**Q22：Docker Compose 里 `localhost` 为什么常不能用？**  
**A：** 容器内 `localhost` 指向 **容器自己**。访问另一容器应使用 **服务名 DNS**（如 `http://gateway:8001`）或 **compose 网络别名**。IM 的 `channels.*_url` 在 compose 中要按此修改。

**Q23：config.yaml 改动后一定要重启吗？**  
**A：** README 说明后端可在 **下次读取配置** 时热更新部分元数据；但 **结构性变更**（端口、compose 服务拓扑等）仍可能需要重启进程或容器，以实际模块为准。
